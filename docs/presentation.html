Introduction

What Is Schedula?

The problem of building university timetables by hand is NP-Hard

The Challenge
  • —Staff availability windows conflict across courses
  • —Room types must match session type (lecture / tutorial / lab)
  • —Student groups must not be double-booked across levels
  • —Soft goals: compress campus days, minimise student travel
Three User Roles
🎓

Coordinator

Manages courses, staff, imports data, configures constraints, triggers generation, reviews and publishes the schedule

👨‍🏫

Staff

Sets weekly availability, views their own teaching schedule for the active term

📚

Student

Views published timetable filtered by their group, level, and enrolled courses

Scheduling Engine

Constraint System

Hard constraints are never violated — soft constraints are minimized by the SA phase

Hard Constraints

HC-1No two sessions share a room at the same time
HC-2No staff member teaches two sessions simultaneously
HC-3No group attends two lectures at the same time
HC-4No subgroup double-booked (tutorial / lab)
HC-6Room type must match session type — halls / tutorial rooms / labs are separate pools
HC-8Session duration must not extend past the last daily period
HC-10Sessions only placed on days when instructor is available

Soft Constraints

SC-1Compress student subgroup sessions onto fewer campus days
SC-2Minimize day-spread per subgroup — target ≤ 4 active days per week
SC-3Prefer slots that cluster with the subgroup's existing assignments
Availability Relaxation

If strict staff availability produces empty domains → solver retries with all campus working days → flags relaxed: true in level stats

Scheduling Algorithm

Solver — Overview

3-phase constraint satisfaction solver in pure JavaScript — no external dependencies

Input
  • —Courses (level, session types, groups, staff)
  • —31 rooms categorized by type
  • —Staff availability per day per term
  • —Institution settings (days, periods, start time)
Output
  • —Conflict-free timetable entries per level
  • —Each entry: course, session type, room, staff, day, period, start/end
  • —Per-level stats: assigned, success, relaxed, over-target count
Key Design Decisions
  • —Runs per level independently — levels don't block each other
  • —Shared global slot index threads through levels — cross-level staff conflicts detected
  • —Time-bounded SA phase — deterministic max runtime, not iteration count
  • —Graceful degradation via availability relaxation fallback
Time Grid

Configurable per institution — daily start, period duration, number of periods, working days. Default: 6 days × 10 periods = 60 time slots × 31 rooms = 1,860 candidate slots per session

Scheduling Algorithm

Solver — 3 Phases

Greedy assigns, Repair fixes, Simulated Annealing optimizes soft constraints

01

Greedy Assignment

backtrack.js · O(n·d)
  • Sort sessions by domain size ASC — most constrained first
  • Then by type: lecture → tutorial → lab
  • For each session: filter all conflict-free slots, pick lowest soft-cost via scoreCandidate()
  • Unassigned sessions passed to Phase 2
  • O(n log n + n·d) — sub-second for <200 sessions
→
02

Conflict Repair

backtrack.js · swap-based
  • Try direct placement first — slots may have freed up
  • If blocked: find the blocking session
  • Temporarily relocate blocker to one of its alternatives
  • Place original session in freed slot
  • Up to 50 swap attempts per session · 3 random-restart rounds · best result kept
→
03

Simulated Annealing

anneal.js · time-bounded
  • Budget: max(1000ms, sessions×3ms) capped at 4000ms
  • Randomly propose moves to alternative slots
  • Accept worsening moves with P = e−ΔCost/T
  • Temperature cools geometrically
  • Minimizes countOverTarget() — subgroups attending >4 days/week

Scheduling Algorithm

Complexity & Performance

Deterministic runtime budget — solver never blocks the server indefinitely

Complexity
Domain construction O(s × t × r)
Greedy phase O(n log n + n·d)
SA phase time-bounded, not iterations
SA max budget 4,000 ms / level
Real-World Scale
  • —ECU seed dataset: multiple levels, ~50+ sessions per level
  • —Full schedule generated in < 5 seconds end-to-end
  • —Runs in-process alongside the web server — no worker thread needed at this scale
Cross-Level Staff Tracking

A globalSlotIndex Map threads through all levels. After each level is solved, its assignments are merged in. A professor teaching both Level 2 and Level 3 is never double-booked — the global index is pre-seeded into every level's conflict check.

Graceful Degradation

If strict staff availability produces infeasible domains (empty slot lists), the solver automatically retries using all campus working days. Relaxed levels are flagged in job stats. No infeasible result is ever returned — the solver always produces the best possible output.
